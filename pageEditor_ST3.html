Feedback & Programme Directory

Technical Description

The Feedback and Programme Directory page (feedback.html) provides users with an interactive way to explore SDG 6-related programmes and submit feedback. It uses the shared HTML template including the header, navigation bar, main content, and footer, and links to the global style.css with additional page-specific embedded CSS.

HTML Structure: The page is divided into three clearly separated sections — the scrollable programme directory, the ratings and reviews summary, and the feedback form. The scrollable programme directory uses a <div class="directory-scroll-wrapper"> with max-height: 540px and overflow-y: auto to allow browsing without taking over the full page. A styled <select> dropdown provides a sort/filter control. Programme cards are built as <article> elements inside a CSS grid container. The feedback form uses <fieldset> and <legend> to group fields into logical sections — Personal Details and Feedback Details — as required by the specification. Form fields include type="text", type="email", type="tel", type="date", type="datetime-local", a <textarea> with maxlength="500", and a <select> dropdown.

CSS Techniques: The programme cards grid uses display: grid; grid-template-columns: repeat(auto-fill, minmax(240px, 1fr)) for a fully responsive layout. Card hover effects use transform: translateY(-6px) and box-shadow transitions for a lift effect. The reviews grid uses the same auto-fill pattern. The custom scrollbar is styled using ::-webkit-scrollbar and ::-webkit-scrollbar-thumb pseudo-elements with the site's teal colour. Form input focus states use border-color transitions for clear visual feedback. Error messages use display: none toggled to display: block by JavaScript.

JavaScript: Three JavaScript systems power this page. First, the programme modal system stores all programme data in a programmeData array and uses openProgramModal(index) to populate and display a modal overlay with the correct image, title, tag, description, and detail for the clicked card. The modal closes via a close button, clicking the backdrop, or pressing Escape. Second, the review modal system uses a reviewData array and openReviewModal(index) to display full review text in a popup. Third, the form validation system uses validateField() to check each required field on submit, displaying inline error messages near the relevant fields. A live character counter on the message textarea uses an input event listener to update remaining characters dynamically, turning red when fewer than 50 characters remain.

Images used in the Feedback page:

  • River Clean-Up Brigade — sourced from River Cleanup organisation (river-cleanup.org)
  • Discussing at University — sourced from An Educator blog (aneducator.com)
  • Cleaning River — sourced from Euronews environment section (euronews.com)
  • Programme card images — sourced from Unsplash (royalty-free) (unsplash.com)

Accessibility

All form fields include clearly associated <label> elements using matching for and id attributes. The <fieldset> and <legend> groupings provide semantic structure for screen readers. Error messages use aria-describedby to associate them with their relevant input fields. The confirmation message uses aria-live="polite" so screen readers announce successful submission automatically. All programme card images include descriptive alt text. The modal overlay traps focus on open using .focus() on the close button, and closes on Escape key press for full keyboard accessibility. The sort dropdown includes a visible <label> for screen reader support.

Team Page

Technical Description

The Team Page (team.html) introduces all four members of the AquaAction group in an interactive, visually engaging layout. It uses the shared HTML template and global style.css, with page-specific embedded CSS in a <style> block scoped entirely to this page.

HTML Structure: The main content area contains a <div class="team-grid"> with role="list" for semantic screen reader support. Each team member is represented by a <div class="member-block"> with tabindex="0" for keyboard accessibility and role="listitem" with a descriptive aria-label. Inside each block, a .member-thumb div contains the profile image, a .member-label div shows the name and role below the thumbnail, and a .member-details div contains the expandable information panel with a heading, role badge, pages list, and contribution summary paragraph.

CSS Techniques: The team grid uses display: grid; grid-template-columns: repeat(auto-fill, minmax(220px, 1fr)) for a responsive four-column layout that collapses to two columns on small screens using a @media (max-width: 600px) breakpoint. The hover and focus expand interaction is implemented entirely in CSS using .member-block:hover .member-details and .member-block:focus .member-details selectors. The .member-details panel uses max-height: 0; overflow: hidden; opacity: 0 as its default state, transitioning to max-height: 380px; opacity: 1 on hover or focus — creating a smooth reveal without JavaScript. The thumbnail uses aspect-ratio: 1/1 for a perfect square crop with object-fit: cover. The keyboard focus ring applies a box-shadow: 0 0 0 4px var(--secondary-blue) outline for visible focus indication.

Note on JavaScript: The Team Page interaction is implemented entirely using CSS :hover and :focus pseudo-classes as permitted by the coursework specification section 4.7, which states the expandable details may be implemented using CSS.

Images used in the Team Page: All four profile images are stored locally in the images/ folder and sourced from Unsplash under a free licence:

Accessibility

Each .member-block includes tabindex="0" so keyboard users can Tab to each card and trigger the :focus reveal of the details panel — no mouse required. Each block also has a descriptive aria-label explaining who the member is and how to interact. The .member-details panel uses aria-hidden="true" since it is purely visual supplementary information and does not need to interrupt screen reader flow. All profile images include meaningful alt text. The team grid uses role="list" and each member block uses role="listitem" for correct semantic list structure. The focus ring is clearly visible using a box-shadow outline in the site's secondary blue colour.

Community Action Content Page (ST3)

Technical Description

The Community Action content page (content_ST3.html) is Student 3's individual content page exploring how communities can take collective action to support SDG 6 — Clean Water and Sanitation. It uses the shared HTML template and global style.css with page-specific embedded CSS in a <style> block.

HTML Structure: The page is divided into clearly defined <section> elements each with a unique id used as anchor targets. An internal navigation menu near the top provides anchor links using href="#section-id" wrapped in an <ul> list styled as pill buttons. Headings follow a strict hierarchy — one <h1> for the page title, <h2> for each section, and <h3> for subsections. A fixed Go to Top button uses position: fixed with an anchor link to href="#top" — implemented in CSS only with no JavaScript. At least four relevant images are included throughout the page with descriptive alt text and CSS styling including borders, shadows, and responsive sizing.

Images used in Content ST3: Three images are stored locally in the images/ folder — river_cleanup.png (River Cleanup org), discussing_atuni.jpg (An Educator blog), and cleaning_river.png (Euronews). Three additional images are loaded directly from Unsplash URLs embedded in the HTML (photo-1531206715517, photo-1488521787991, photo-1559825481) — all free to use under the Unsplash licence.

CSS Techniques: The internal navigation uses display: flex; flex-wrap: wrap; gap: 0.5rem for a responsive pill button row. Content sections use scroll-margin-top: 90px to prevent headings from hiding behind the sticky header when navigating via anchor links. Section headings use a border-bottom: 3px solid var(--secondary-blue) design for clear visual separation. Images are styled with border-radius, box-shadow, and max-width: 100% for responsive sizing. The Go to Top button uses position: fixed; bottom: 2rem; right: 2rem to remain visible as the user scrolls.

Accessibility

The internal navigation uses aria-label="Page sections" to distinguish it from the main navigation bar. All images include descriptive alt text relevant to the content they illustrate. The heading hierarchy is logically ordered from <h1> through <h3> to support screen reader navigation. The Go to Top button includes a descriptive aria-label and title. Colour contrast between body text and backgrounds meets WCAG AA standards throughout using the site's dark abyss colour on light backgrounds.

Challenges and Lessons Learned

Challenge 1 — Duplicate JavaScript functions: During development of the feedback page, two versions of openProgramModal() and openReviewModal() were accidentally defined in the same script block, causing the second definition to silently override the first. This was resolved by removing the duplicate function declarations and consolidating the logic into a single definition for each function. This taught me to carefully manage JavaScript function scope in single-page scripts.

Challenge 2 — CSS-only hover expand on Team Page: Initially the expandable details panel did not animate smoothly because max-height was set to auto, which CSS cannot transition. Replacing auto with a fixed pixel value of 380px allowed the transition to work correctly. This taught me that CSS transitions require explicit numeric values to interpolate between.

Challenge 3 — Form validation with HTML5 and JavaScript: Combining HTML5 required attributes with custom JavaScript validation caused double error messages in some browsers. The solution was to call e.preventDefault() on form submit and handle all validation in JavaScript, using HTML attributes only as a fallback. Redundant aria-required="true" attributes were also removed since they are unnecessary when required is already present.

Challenge 4 — Empty heading validation warnings: The modal headings (<h2 id="modal-title">) were flagged as empty heading warnings by the W3C validator because they are populated dynamically by JavaScript. These are acceptable warnings that do not prevent validation, as the headings receive content at runtime when a modal is opened.

Lessons Learned: This project deepened my understanding of JavaScript event-driven programming and the importance of keeping functions organised and non-duplicated. I also learned that CSS-only interactions can be powerful and sufficient for simpler UI patterns like hover reveals, reducing reliance on JavaScript where it is not required.
